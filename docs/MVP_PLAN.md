# LayLingo MVP 기획서

## 1. 제품 목표

사용자가 텍스트형 PDF를 올리고, 화면에서 문장을 클릭하면 즉시 한국어 번역을 요청할 수 있다. 번역 결과는 원본 위치 위에 표시되며 폰트, 크기, 색상, 굵기, 정렬을 편집할 수 있다.

## 2. MVP 성공 기준

- 로컬 PDF 업로드와 페이지 렌더링
- 페이지별 텍스트 영역 추출 및 클릭 선택
- 클릭한 텍스트를 서버 API로 전송하고 번역 결과 표시
- 번역문 직접 수정
- 폰트, 크기, 색상, 굵기, 정렬 실시간 변경
- 페이지 이동과 확대/축소에서도 오버레이 좌표 유지
- 처리 중, 실패, 데모 모드를 사용자가 구분 가능
- 새로고침 전까지 편집 상태 유지

## 3. 이번 범위에서 제외

- 스캔 PDF OCR
- 원본 텍스트 제거 및 최종 PDF 바이너리 내보내기
- 사용자 폰트 업로드와 서버 임베딩
- 다중 사용자·로그인·클라우드 저장
- 표 셀 및 문단 자동 병합

위 항목은 화면 편집 좌표와 번역 품질이 검증된 다음 단계에서 구현한다. 특히 PDF 내보내기는 원본의 그래픽·폰트 손상을 피하기 위해 별도 백엔드 렌더링 검증이 필요하다.

## 4. 사용자 흐름

1. PDF 파일을 선택하거나 드롭한다.
2. 문서의 첫 페이지와 클릭 가능한 텍스트 영역이 표시된다.
3. 문장을 클릭하면 선택 상태가 되고 번역을 요청한다.
4. 오른쪽 패널에서 번역문과 서식을 조정한다.
5. 번역 결과가 PDF 화면 위에서 즉시 갱신된다.
6. 다른 문장을 선택해 같은 작업을 반복한다.

## 5. 기술 설계

- 프론트엔드: React, TypeScript, Vite
- PDF 표시·추출: PDF.js (`pdfjs-dist`)
- 번역 서버: Express + OpenAI Responses API
- 검증: Zod 요청 검증, TypeScript 빌드, Vitest

PDF.js의 viewport 변환값을 텍스트 항목의 transform과 결합해 화면 좌표를 구한다. API 키는 브라우저에 노출하지 않고 서버 환경 변수에서만 읽는다.

## 6. 안전 및 품질 원칙

- PDF는 MVP에서 브라우저 메모리 안에서만 읽고 서버로 업로드하지 않는다.
- 번역 API에는 선택한 텍스트와 최소 문맥만 전송한다.
- 빈 문자열, 과도하게 긴 입력, 잘못된 언어 코드는 서버에서 거부한다.
- API 키가 없으면 데모 모드임을 응답과 UI에 명시한다.
- 색상 외에도 테두리와 텍스트로 선택·오류 상태를 표현한다.
